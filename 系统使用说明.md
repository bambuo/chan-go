# 缠中说禅信号分析系统 — 使用说明

## 1. 系统概述

本系统是一个基于缠中说禅理论的**实时多级别信号分析引擎**。它作为黑盒计算引擎，通过 Redis Stream 接收 K 线数据，实时计算缠论结构，产出标准化的买卖点信号。

### 核心流程

```
K 线数据 → Redis Stream (chan:klines:{symbol})
              ↓
          M1 输入网关 (XREADGROUP)
              ↓
          M2 缠论 Pipeline (合并→分型→笔→线段→中枢→走势类型)
              ↓
          M3 结构树 (版本化存储)
              ↓
          M4 递归级别 (L1→L2→L3→L4)
              ↓
          M5 信号引擎 (三类买卖点)
              ↓
          M6 共振引擎 (区间套/跨层共振)
              ↓
          M8 输出网关 → Redis Stream (chan:signals:{symbol}) + REST API
```

## 3. 启动引擎

### 3.1 编译

```bash
cd chan-go
go build -o bin/server ./cmd/server/
```

### 3.2 快速启动（开发调试）

```bash
go run ./cmd/server/ --symbols TEST --log-level debug
```

### 3.3 生产启动

```bash
./bin/server \
  --symbols BTCUSDT,ETHUSDT,SOLUSDT \
  --log-level info \
  --redis-addr localhost:6379 \
  --http-addr 0.0.0.0 \
  --http-port 8080 \
```

### 3.5 安全关闭

按 `Ctrl+C` 触发 SIGINT，系统会依次关闭：

1. M8 HTTP 服务（5 秒超时）
2. M1 输入网关（停止 XREADGROUP）
3. M6 共振引擎
4. 释放资源

## 4. 输入数据 — 推送 K 线

### 4.1 数据格式

引擎消费 Redis Stream `chan:klines:{symbol}`。每条消息必须包含以下字段：

```
XADD chan:klines:TEST * \
  symbol TEST \
  ts 1712345678000 \
  open 100.0 \
  high 105.0 \
  low 99.0 \
  close 104.0 \
  volume 1234.5
```

### 4.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `symbol` | string | 是 | 交易对（与 stream key 中的 symbol 一致） |
| `ts` | int64 | 是 | K 线开盘时间，Unix 毫秒时间戳 |
| `open` | decimal | 是 | 开盘价 |
| `high` | decimal | 是 | 最高价 |
| `low` | decimal | 是 | 最低价 |
| `close` | decimal | 是 | 收盘价 |
| `volume` | decimal | 否 | 成交量 |

### 4.3 数据要求

- **时间戳**：必须是非未来时间（引擎会校验 `OpenTime` 是否远超当前时间）
- **连续性**：推荐按时间顺序推送，引擎会检测时间缺口并记录指标
- **最低数据量**：至少需要 7 根 K 线才能形成分型，约 15 根以上才能形成笔和初步结构

### 4.4 注入示例

```bash
# 直接使用 redis-cli
redis-cli XADD chan:klines:BTCUSDT * \
  symbol BTCUSDT ts $(date +%s%3N) \
  open 50000 high 51000 low 49800 close 50500 volume 100.5

# 或用 Python 批量注入
python3 -c '
import subprocess, time
base_ms = int((time.time() - 900) * 1000)
klines = [
    (100,102,99,101), (101,105,100,104), (104,108,103,107),
    (107,110,106,109), (109,112,108,111), (111,113,110,112),
    (112,114,105,106), (106,108,103,104), (104,106,100,101),
    (101,103,98,99),  (99,100,95,96),    (96,98,94,97),
    (97,102,96,101),  (101,106,100,105), (105,109,104,108),
]
for i,(o,h,l,c) in enumerate(klines):
    ts = base_ms + i * 60000
    subprocess.run(["redis-cli","XADD","chan:klines:TEST","*",
        "symbol","TEST","ts",str(ts),
        "open",str(o),"high",str(h),"low",str(l),"close",str(c),"volume","1000"])
'
```

## 5. 输出数据 — 消费信号

### 5.1 Redis Stream 输出

引擎将信号写入 `chan:signals:{symbol}`。每条消息格式：

```json
{
  "type": "signal.created",
  "ts": "1712345678000",
  "data": {
    "signalId": "...",
    "symbol": "BTCUSDT",
    "type": "BUY_1",
    "level": "L1",
    "state": "candidate",
    "confidence": 0.75,
    "price": 50123.45,
    "anchor": { ... },
    "targets": { ... },
    "resonance": { ... },
    "evidence": { ... }
  }
}
```

**事件类型**：

| type | 说明 |
|------|------|
| `signal.created` | 新信号产生（状态：candidate） |
| `signal.stateChanged` | 信号状态变更（confirmed/invalidated/superseded） |
| `resonance.triggered` | 共振事件，附有共振详情 |

### 5.2 消费方式

```bash
# 终端调试
redis-cli XREAD GROUP chan-engine consumer-name \
  COUNT 10 BLOCK 5000 \
  STREAMS chan:signals:BTCUSDT >

# 程序消费（Go 示例）
client.XReadGroup(ctx, &redis.XReadGroupArgs{
    Group:    "my-group",
    Consumer: "my-consumer",
    Streams:  []string{"chan:signals:BTCUSDT", ">"},
    Count:    10,
    Block:    0,
})
```

## 6. 信号系统详解

### 6.1 六类买卖点

| 类型 | 名称 | 触发条件 |
|------|------|---------|
| `BUY_1` | 第一类买点 | 下跌趋势背驰段出现底背驰 |
| `BUY_2` | 第二类买点 | 一买后回调不破前低（次级别底背驰） |
| `BUY_3` | 第三类买点 | 突破中枢后回调不进入中枢区间 |
| `SELL_1` | 第一类卖点 | 上涨趋势背驰段出现顶背驰 |
| `SELL_2` | 第二类卖点 | 一卖后反弹不破前高（次级别顶背驰） |
| `SELL_3` | 第三类卖点 | 跌破中枢后反弹不进入中枢区间 |

### 6.2 信号状态机

```
candidate ──→ confirmed ──→ superseded
                  │
                  └──→ invalidated
```

- **candidate**：初步识别，等待确认
- **confirmed**：条件满足，正式生效
- **invalidated**：结构变化导致失效
- **superseded**：被更新的同类型信号取代

### 6.3 信号等级

每个信号包含：

- **confidence**（置信度，0–1）：基于证据强度的量化评分
- **strength**（强度）：基于动能指标的力度评分
- **recastRisk**（漂移风险）：结构变更可能性
- **targetPrice**（目标位）：结构性参考位
- **invalidationPrice**（失效位）：触发失效的价位

## 7. 共振机制

### 7.1 G-2 区间套

当高级别（如 L2）走势类型与低级别（如 L1）信号同向，且低级别信号价格区间落在高级别走势范围内时，触发区间套共振。

**效果**：提升信号置信度，`confidence` 乘以 `(1 + 0.1 × nestingDepth)`

### 7.2 G-1 跨层共振

当多个级别同时出现同向信号时触发。

**效果**：每个额外级别的同向信号贡献 0.05 的 confidence 加成

### 7.3 A3 方向过滤

信号方向与主要级别的趋势方向对齐时获得加成。

**效果**：
- 方向一致：`boost ∈ [0, 0.2]`
- 方向不一致：`boost = 0`

## 8. 快照与故障恢复

### 8.1 自动快照

引擎按 `--snapshot-period` 间隔（默认 5 分钟）自动创建快照，包含：

- 当前信号的 Redis offset
- 活跃信号列表
- 双轨状态

### 8.2 恢复流程

重启时引擎自动从最近的快照恢复：

1. 读取 `data/snapshots/` 目录下的最新快照
2. 恢复 Redis 消费组游标（`XGROUP SETID`）
3. 恢复内存中的信号状态
4. 从快照处继续消费 K 线

### 8.3 快照保留

`--snapshot-retain` 控制保留的快照数量（默认 24 个），超出部分自动清理。

## 10. 常见问题

### 10.1 引擎启动报错 "address already in use"

端口被占用，先释放：

```bash
lsof -ti:8080 | xargs kill -9
```

### 10.2 引擎报错 "NOGROUP No such key"

重启引擎时 Redis Stream 或消费组不存在。清理旧数据后重启：

```bash
redis-cli DEL chan:klines:TEST
# 重启引擎（会自动创建消费组和 Stream）
go run ./cmd/server/ --symbols TEST
# 然后重新注入 K 线
```

### 10.3 结构数据为空

可能原因：
1. K 线数据量不足（需要至少 7 根 K 线形成分型）
2. K 线缺少 `symbol` 或 `ts` 字段
3. Redis 消费组游标已越过数据

### 10.4 引擎不产生信号

信号生成需要足够的缠论结构（笔→中枢→走势类型），通常需要：
- L1 级别：30+ 根 K 线才能形成完整的中枢+走势类型
- 更高层级需要更多数据

## 11. 最佳实践

### 11.1 数据源集成

```python
# Python 示例：推送交易所 K 线到引擎
import redis
r = redis.Redis()

def push_kline(symbol, kline):
    r.xadd(f"chan:klines:{symbol}", {
        "symbol": symbol,
        "ts": int(kline["openTime"]),
        "open": str(kline["open"]),
        "high": str(kline["high"]),
        "low": str(kline["low"]),
        "close": str(kline["close"]),
        "volume": str(kline["volume"]),
    })
```

### 11.2 信号消费端

```python
# Python 示例：消费信号流
r = redis.Redis()
group = "trading-bot"
consumer = "bot-1"

# 创建消费组（首次）
try:
    r.xgroup_create("chan:signals:BTCUSDT", group, id="0", mkstream=True)
except:
    pass

while True:
    msgs = r.xreadgroup(group, consumer,
                        {"chan:signals:BTCUSDT": ">"}, count=10, block=5000)
    for stream, messages in msgs:
        for msg_id, msg in messages:
            print(f"信号: {msg[b'type']} → {msg[b'data']}")
```

### 11.3 生产部署建议

- 使用 Redis 集群或多副本部署提高可用性
- 设置 `--snapshot-period` 不超过 300 秒
- 日志级别生产环境设为 `info`
- 监控 Prometheus 指标中的 `kline_delay_seconds`
