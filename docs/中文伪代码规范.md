# 中文伪代码规范

本文档定义了如何用中文精确描述算法逻辑。核心原则是：**用中文自然语言说清楚，不需要编程语言的结构。

---

## 1. 文档结构

每个算法单元按以下结构组织：

```
## 算法名称

输入：
- 参数名：说明。
- 参数名：说明。

输出：
- 说明。

算法：
  用中文描述算法逻辑。
```

输入和输出用精确的数据描述，算法部分用自然中文叙述。

---

## 2. 叙述方式

规范的优先顺序是：自然叙述 > 结构化伪代码。

如果一个算法可以用一段话描述清楚，就写一段话。如果用结构化方式更方便，也可以混合使用。

### 2.1 计算和条件

用中文描述计算和判断逻辑：

```
若 合并K线数量 < 3，则直接返回空列表。

对每相邻三根合并K线：
  若中间一根的高 > 两侧的高 且 中间一根的低 > 两侧的低，
  则该合并K线为顶分型，加入顶分型列表。
```

条件判断使用：

| 写法 | 含义 |
| --- | --- |
| `若 ... 则` | if |
| `否则若` | else if |
| `否则` | else |
| `且` | AND |
| `或` | OR |
| `非` | NOT |

### 2.2 循环和遍历

遍历可以用自然叙述表达：

```
对 笔列表 中每条笔，若其方向与当前线段方向相反则加入特征序列。
```

或者用表格化的方式表达重复操作。

### 2.3 比较和运算

比较运算保留精确符号：

| 写法 | 含义 |
| --- | --- |
| `==` | 等于 |
| `!=` | 不等于 |
| `>` | 大于 |
| `<` | 小于 |
| `>=` | 大于等于 |
| `<=` | 小于等于 |

数学运算也保留符号：

```
面积缩小 5% 以内才算背驰：当前面积 < 前面积 × 0.95
跨度 = 终点索引 - 起点索引
```

### 2.4 数据结构操作

对数组/列表的操作：

| 写法 | 含义 |
| --- | --- |
| `列表[索引]` | 取第 索引 个元素 |
| `列表[最后]` | 取最后一个元素 |
| `列表[0:-2]` | 取前 n-2 个元素 |
| `列表.追加(元素)` | 末尾添加 |
| `列表.弹出()` | 取出最后一个 |
| `长度(列表)` | 元素个数 |

### 2.5 值和类型

| 写法 | 含义 |
| --- | --- |
| `真` | true |
| `假` | false |
| `空` | null/nil |
| `报错` | 抛出异常 |

## 3. 算法描述的三种风格

### 3.1 叙述风格（优先）

适合简单或中等复杂度的逻辑，一段话可以讲清楚。

> **示例：包含处理**
>
> 从第一根K线开始，依次处理后面的每根K线：
>
> 若当前K线与前一根存在包含关系（当前高 ≤ 前一根高 且 当前低 ≥ 前一根低，或其逆），
> 则按当前方向合并：
>   - 向上方向，取两边的高中更高、低中更高；
>   - 向下方向，取两边的高中更低、低中更低。
> 合并后的K线替换前一根，继续与下一根比较。
>
> 若不存在包含关系，则当前K线直接加入结果序列。
>
> 方向判定：当前K线高点高于前一根则方向向上，低点低于前一根则方向向下。

### 3.2 混排风格（推荐）

条件分支较多的逻辑，叙述为主，分支用结构化方式。

> **示例：确认笔更新**
>
> 每次计算前先清理上一轮的虚笔。
>
> 对输入的合并K线：
>   - 若无分型，检查最终K线索引是否有变化即可。
>   - 若 笔列表为空，尝试用该合并K线创建第一笔。
>   - 若 该合并K线与最后终点同类型，则尝试更新最后一笔的终点。
>   - 否则若 该合并K线可与最后终点成笔，则添加新笔，并将该合并K线设为新的最后终点。
>   - 否则若 可触发更新峰值修正，则执行修正。

### 3.3 结构化风格（必要时使用）

适合复杂的多分支条件判定。

> **示例：分型有效性检查（顶分型起点）**
>
> 起点为顶分型，候选终点为底分型。
>
> 方法选择：
>   - `半`（默认）：看候选终点前一根和起点后一根。
>   - `宽松`：只看两个分型自身。
>   - `严格` 或 `完全`：看两个分型三元素。
>
> 计算规则：
>   - 若方法为 `半`：
>     第二元素高点 = 最大值(第二元素.前一根.高, 第二元素.高)
>     自身低点 = 最小值(自身.低, 自身.下一根.低)
>   - 若方法为 `宽松`：
>     第二元素高点 = 第二元素.高
>     自身低点 = 自身.低
>   - 若方法为 `严格` 或 `完全`（虚笔时只取两根，否则取三根）：
>     第二元素高点取前一根和自身（及后一根）中的最高值
>     自身低点取前一根和自身（及后一根）中的最低值
>
> 若方法为 `完全`，要求自身低点 > 第二元素高点；
> 否则要求 自身高点 > 第二元素高点 且 第二元素低点 < 自身低点。

## 4. 命名规则

### 4.1 算法/函数

用中文动宾短语表达，表达"做什么"，不表达"怎么做"。

| 名称 | 说明 |
| --- | --- |
| `包含处理` | 对原始K线做包含处理 |
| `识别分型` | 在合并K线上识别顶底分型 |
| `更新确认笔` | 根据新分型更新确认笔状态 |
| `可否成笔` | 判断两个分型是否可以成笔 |
| `终点是否峰值` | 检查笔终点是否为区间极值 |
| `尝试创建第一笔` | 尝试从候选分型中创建第一笔 |

### 4.2 数据对象

数据对象名使用领域术语，不自造新词。

| 名称 | 说明 |
| --- | --- |
| `合并K线` | 包含处理后的K线 |
| `笔列表` | 已生成的笔序列 |
| `最后终点` | 最后一笔的终点分型 |
| `候选分型列表` | 第一笔生成前缓存的候选分型 |
| `合并K线.高` | 合并K线的最高价 |
| `合并K线.低` | 合并K线的最低价 |

### 4.3 配置

```
配置.字段名
```

| 名称 | 含义 |
| --- | --- |
| `配置.笔算法` | 笔算法模式 |
| `配置.是否严格` | 是否使用严格跨度 |
| `配置.笔分型检查` | 分型有效性检查方式 |

### 4.4 领域缩写

通用技术指标缩写不翻译（这类缩写不依赖编程语言，而是行业标准符号）：

```
MACD, RSI, BOLL, MA, EMA
```

---

## 5. 编写原则

1. **自然叙述优先**。一段话能讲清楚的，不套编程结构。
2. **精确定义边界**。条件、临界值、数量关系必须精确，不能模糊。
3. **结构化用于复杂分支**。当逻辑分支多于 3 个时，可以用结构化方式（如上面"分型有效性检查"的例子）。
4. **名称即注释**。变量名、算法名用中文领域术语，读者看到名字就明白含义。
5. **数据定义与算法分离**。算法正文前先说明输入输出的数据结构和字段含义。
6. **避免层级嵌套**。嵌套超过 3 层时，用自然分段代替缩进。
7. **数值条件明确**。阈值、比例、边界值必须数字写清。
8. 领域通用缩写（MACD、RSI、BOLL 等）保留英文不翻译。
9. **读者读完就能实现**。这是最终的检验标准。
